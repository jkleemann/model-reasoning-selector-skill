# WEBVP-35648 Spring Boot 4 Line Split Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Split `web-vp-commons` into a temporary Spring Boot 3 maintenance branch that continues the existing `3.x` versioning scheme and a Spring Boot 4 `main` line that starts at `4.0.0`, then make the full Maven reactor green for the first `4.x` release.

**Architecture:** First establish release-line infrastructure while the code is still Boot 3 compatible, including full Release Publish Path parity for maintenance branch `sb3-maintenance`; then create `sb3-maintenance`; then migrate `main` reactor-wide to Boot 4. The maintenance branch name describes purpose, while semantic-release maps it to the existing `3.x` version range. Do not create a synthetic `3.x` no-op release solely as a publish proof; use workflow inspection and semantic-release dry-run before Boot 4 migration, then let the first real Spring Boot 3 maintenance fix prove live artifact publication. Dependency management moves to the root POM; modules consume managed Spring Boot, Spring Cloud, Spring Framework, Spring Security, and Jakarta versions without local Boot variants. The web-vp-commons artifact version and the upstream Spring Boot Dependency Baseline are distinct: web-vp-commons `4.0.0` may use a later approved Spring Boot `4.0.x` patch.

**Hotfix policy:** Keep `hotfix/*` as the existing working branch pattern. The PR target decides the Release Line: merge to `sb3-maintenance` for `3.x.y`, merge to `main` for `4.x.y` after the Boot 4 migration. Do not add a separate hotfix Release Line.

**Maintenance activation:** Cut and verify `sb3-maintenance` before the Boot 4 migration. After the branch exists, real Spring Boot 3 fixes should target `sb3-maintenance`; if an urgent Spring Boot 3 fix is needed before the branch cut, land it on the still-Boot-3-compatible `main` and include it in the branch cut. Do not create a no-op maintenance release solely to prove publication.

**Research alignment:** Release-pipeline and Boot 4 migration research is captured in `docs/superpowers/research/2026-06-03-webvp-35648-release-and-boot4-research.md`. A full tag fetch corrected the release model: `origin/main` is already at `3.7.5`, so `sb3-maintenance` must continue `3.x` and `main` must become `4.x`. The remaining release-line risks are validation gates: semantic-release dry-run on the real maintenance branch, actual workflow run inspection, and first real maintenance publish proof.

**Tech Stack:** Maven multi-module reactor, Kotlin, Java 21, Spring Boot, Spring Cloud OpenFeign, springdoc-openapi, semantic-release, GitHub Actions.

# Spring Boot 4 Dependency Management Rationale

## Policy

The Spring Boot 4 line centralizes Spring-related dependency management in the root `pom.xml`.

Module POMs must not pin Spring Boot, Spring Framework, Spring Security, Jakarta, or Jackson 2 versions unless the pin is listed in this document with a reason.

## Managed Lines

- web-vp-commons artifact line: `4.x.y`
- Spring Boot Dependency Baseline: `4.0.6`
- Spring Cloud: `2025.1.0`
- springdoc-openapi: `3.0.3`
- Java: `21`

## Allowed Explicit Pins

No Spring Boot, Spring Framework, Spring Security, Jakarta, or Jackson 2 explicit module-level pins are allowed in the initial migration unless listed below.

## Jackson Decision

The Spring Boot 4 line uses a Jackson 3-first strategy. Source code, generated code, and dependency management should align with the Jackson version managed by the Spring Boot 4 Dependency Baseline.

Do not keep a root Jackson 2 BOM or `jackson.version` pin by default.

A Jackson 2 pin is allowed only for:

- public `web-vp-commons` API compatibility
- OpenAPI generator or generated model constraints
- third-party DVAG or external dependency constraints

Every Jackson 2 exception must include:

- exact dependency and version
- affected module
- reason
- removal condition
