## Authoritative Module Definitions

The module definitions for PR B are the 23 files under `docs/spring-boot-4/module-specs/`. Treat those files as the authoritative per-module contract. Do not infer module ownership only from the root plan or from a broad `pom.xml` search.

Every module spec contains:

- `Classification`: `boot-direct`, `spring-adjacent`, `test-support`, or `reactor-only`.
- `Evidence`: dependencies, source paths, tests, and generated-code surfaces that make the module Boot 4 relevant.
- `Migration Rules`: module-local dependency and generation constraints.
- `Consumer Dependency Expectation`: what consuming applications must provide and what the module intentionally exposes.
- `Required Spring Boot 4 Tests`: the exact focused Maven command for that module.
- `Module-Specific Proof Requirements`: extra checks such as OpenFeign, JPA, Jackson persistence, generated OpenAPI source, or auto-configuration proof.
- `Completion Evidence`: dependency-tree, mixed-generation, scope-bleed, and module-command evidence required before marking that module done.

Implementation rule for a less-contextual agent:

1. Open the module spec before touching that module.
2. Copy the module's `Required Spring Boot 4 Tests` command into the work log.
3. Make only the POM/source/test/docs changes needed for that module's spec.
4. Run the module command.
5. Update the module spec completion evidence only after the command result is known.
6. Commit the module only in the matching batch commit from this plan.

import jakarta.persistence.Converter
